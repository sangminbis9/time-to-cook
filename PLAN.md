## 0. 이 문서의 목적

이 문서는 Claude Code의 `/plan` 모드에 그대로 제공할 수 있는 **전체 프로젝트 제작 명세서이자 구현 로드맵**이다.

Claude Code는 이 문서를 읽은 뒤 바로 대규모 구현을 시작하지 말고, 먼저 현재 저장소의 구조와 기존 Unity 코드를 감사한 다음 아래 요구사항에 맞는 구체적인 단계별 계획을 작성해야 한다.

이 프로젝트는 기존 3D Overcooked형 프로토타입을 단순 확장하는 작업이 아니다.  
최종 목표는 다음과 같다.

> **스타듀밸리 계열의 따뜻한 생활형 힐링 도트 그래픽을 사용하는, 탑다운 2D 기반의 온라인 2인 협동 요리 프랜차이즈 타이쿤 게임**

핵심은 다음 네 요소를 하나의 일관된 게임으로 결합하는 것이다.

1. Overcooked처럼 직접 움직이며 요리하는 실시간 협동 플레이
2. 매장·직원·재고·시장·대출을 관리하는 경영 시뮬레이션
3. 여러 도시와 국가에 매장을 확장하는 프랜차이즈 성장
4. 바쁘지만 화면은 편안하고 따뜻한 힐링 도트 감성

# 1. Claude Code 작업 원칙

## 1.1 먼저 해야 할 일

Claude Code는 구현 전에 반드시 다음을 수행한다.

1. 현재 Unity 프로젝트 전체 구조를 조사한다.
2. 모든 C# 스크립트와 주요 씬·프리팹·ScriptableObject 구조를 확인한다.
3. 기존 3D 프로토타입 코드 중 재사용 가능한 로직과 폐기해야 할 로직을 분리한다.
4. 이 문서의 요구사항과 현재 구현 사이의 차이를 정리한다.
5. 시스템 간 의존성을 분석한다.
6. 구현 순서를 위험도가 낮은 단계부터 제안한다.
7. 첫 번째 플레이 가능한 수직 슬라이스에 필요한 최소 범위를 정의한다.
8. 각 단계의 완료 조건과 테스트 방법을 명시한다.

## 1.2 바로 하지 말아야 할 일

다음은 계획 단계에서 금지한다.

- 저장소를 충분히 조사하기 전에 대규모 리팩터링 시작
- 기존 코드를 무조건 유지하거나 무조건 전부 삭제
- 네트워크·저장·아이템 권한 구조 없이 UI부터 대량 구현
- 3D 물리 기반 구조를 그대로 억지로 2D 게임에 사용
- 모든 국가·도시·메뉴·직원을 한 번에 구현
- 미확정된 세부 수치 때문에 계획을 중단
- 사소한 슬롯 수, 가격, 정확한 시간 값을 매번 사용자에게 질문

사소한 기본값은 합리적으로 설정하고, 나중에 ScriptableObject 또는 설정 데이터로 쉽게 조정할 수 있게 만든다.

## 1.3 계획 결과물 형식

Claude Code는 `/plan` 결과에 다음을 포함해야 한다.

- 현재 코드베이스 진단
- 유지/수정/폐기 대상 파일 목록
- 제안 아키텍처
- 씬 구조
- 데이터 구조
- 네트워크 권한 구조
- 저장 구조
- 구현 단계별 작업 목록
- 단계별 테스트 기준
- 예상 위험 요소
- 첫 번째 수직 슬라이스 범위
- 이후 확장 로드맵

# 2. 프로젝트 핵심 정체성

## 2.1 장르

- 탑다운 2D 픽셀 아트
- 온라인 2인 협동
- 싱글플레이 지원
- 실시간 요리 액션
- 프랜차이즈 경영
- 도시·국가 확장
- 직원 관리
- 시장 조사
- 장기 성장형 타이쿤

## 2.2 핵심 플레이 감정

플레이어는 다음 감정을 느껴야 한다.

- 작은 가게를 정성스럽게 운영하는 즐거움
- 둘이 역할을 나눠 협력하는 만족감
- 매장을 꾸미고 성장시키는 성취감
- 새로운 도시를 분석하고 진출하는 전략성
- 바쁘지만 지나치게 공격적이지 않은 몰입감
- 따뜻하고 소박한 세계에 머무는 힐링 감성

## 2.3 차별점

이 게임은 일반적인 Overcooked 복제나 단순 식당 타이쿤이 아니다.

핵심 차별점:

- 플레이어가 직접 요리하는 액션과 장기 경영이 동일한 세이브 안에서 연결됨
- 한 프랜차이즈를 공유하지만 각 플레이어가 다른 매장·도시에서 동시에 활동 가능
- 도시마다 수요·물가·경쟁·관광·경제 상황이 달라 메뉴 전략이 달라짐
- 직원은 단순 버프가 아니라 실제 작업을 독립적으로 수행
- 플레이어 캐릭터마다 고유 특성과 영구 성장 트리가 있음
- 전체 분위기는 생활형 힐링 도트 게임을 유지함

# 3. 아트 디렉션

## 3.1 필수 시각 방향

- **무조건 2D 픽셀 아트**
- **탑다운에 가까운 시점**
- 기본 타일 크기: **32×32 px**
- 스타듀밸리 계열의 생활형 힐링 감성에서 영감을 받되 직접 복제하지 않음
- 따뜻한 나무색, 크림색, 연갈색, 민트, 하늘색, 살구색 중심
- 배경은 편안하게
- 음식과 상호작용 오브젝트는 또렷하게
- 도시별 개성은 있지만 전체 세계관과 팔레트는 통일
- 과도한 네온, 현실적 음영, 무거운 재난 분위기, 차가운 기업형 UI 지양

## 3.2 캐릭터

- 귀여운 SD형 도트 캐릭터
- 역할과 캐릭터를 실루엣·모자·앞치마·색상으로 구분
- 작은 크기에서도 방향과 행동이 명확해야 함
- 이동, 아이템 보유, 조리, 상호작용 상태가 읽혀야 함
- 캐릭터별 개성이 시각적으로 드러나야 함

## 3.3 매장

- 실제 생활감이 있는 작은 가게처럼 보여야 함
- 나무 가구, 식물, 조명, 메뉴판, 소품 적극 활용
- 주방 동선과 상호작용 설비는 명확하게 구분
- 장식이 많아도 플레이 가독성을 방해하면 안 됨
- 벽 정면과 바닥이 함께 보이는 스타듀밸리형 실내 표현
- 사선 아이소메트릭 타일은 사용하지 않음

## 3.4 음식

각 음식은 저해상도에서도 다음 상태가 확실히 구분돼야 한다.

- 생재료
- 손질됨
- 비가열 전처리 완료
- 조리 중
- 정상 완료
- 과조리
- 탄 상태
- 소스 또는 옵션이 적용된 상태

음식은 현실적인 묘사보다 귀엽고 먹음직스러운 표현을 우선한다.

## 3.5 UI

- 픽셀 아트와 어울리는 목재·크림색 패널
- 둥글고 부드러운 인상
- 정보량이 많아도 시각적 구역 구분이 명확해야 함
- 타이쿤 UI가 차갑고 복잡한 ERP 화면처럼 보이면 안 됨
- 시장 정보는 수첩·게시판 느낌
- 도시 선택은 지도·카드 느낌
- 냉장고와 플레이어 인벤토리는 마인크래프트식 슬롯 UI
- 실패 안내 문구는 인벤토리 위에 표시

원래 선정 기준 (충족 확인):

- 호스트 권한 모델 구현 용이
- 씬 이동 및 다중 매장 상태 동기화 가능
- 아이템 중복 획득 방지
- 저장 데이터와 런타임 상태 분리 가능
- 2인 협동에 과도하게 무겁지 않음
- 장기 유지보수 가능

# 5. 전체 게임 루프

게임의 하루는 세 단계로 구성한다.

## 5.1 준비 단계

- 세계 시간 일시정지
- 제한 시간 없음
- 매장별 메뉴 설정
- 재료 주문
- 직원 배치
- 직원 재배치
- 가격 설정
- 광고 설정
- 시장 정보 확인
- 시설 구매·배치
- 벽·바닥·문·계단·가구 수정
- 냉장고 보관 상태 확인
- 연구 구매
- 대출 관리
- 도시 및 신규 매장 진출 검토
- 캐릭터 영구 업그레이드 구매

두 플레이어가 접속 중이면 둘 다 준비 완료해야 영업 시작.  
한 명만 접속 중이면 혼자 시작 가능.

## 5.2 실시간 영업 단계

- 주문 발생
- 플레이어 직접 이동·조리·서빙
- 직원 자동 작업
- 손님 처리
- 장비 및 재난 이벤트 대응
- 냉장고 사용 가능
- 여러 매장이 동시에 시뮬레이션
- 플레이어가 없는 매장은 자동화 상태로 통계 처리

## 5.3 마감 및 정산 단계

- 세계 시간 일시정지
- 제한 시간 없음
- 매출·비용·평판·폐기·주문 결과 표시
- 맵 위 모든 아이템 자동 폐기
- 냉장고 내부의 보관 가능한 아이템만 유지
- 직원 상태 및 다음 날 휴가 확인
- 시장·경제 변화 반영
- 대출 이자 및 비용 처리
- 다음 준비 단계로 이동

## 5.4 오프라인 진행

- 최소 한 명이 접속 중이면 모든 매장 운영 상태 진행
- 두 플레이어 모두 오프라인이면 세계 시간 완전 정지
- 오프라인 수익 없음
- 부재 플레이어 캐릭터는 나타나지 않음
- AI가 부재 캐릭터를 대신 조종하지 않음
- 다른 친구가 부재 캐릭터를 대체하지 않음

# 6. 월드·국가·도시 구조

## 6.1 초기 국가

- 대한민국

## 6.2 초기 대한민국 도시

- 서울
- 부산
- 인천
- 대구
- 대전
- 광주
- 울산
- 제주

## 6.3 초기 일본 도시

- 도쿄
- 요코하마
- 오사카
- 나고야
- 삿포로
- 후쿠오카
- 교토
- 고베

## 6.4 시작 도시

- 시작 도시는 **인천**
- 인천은 최초 시장 정보를 최고 수준으로 무료 제공
- 이후 갱신은 일반 규칙에 따라 비용 발생
- 시작 메뉴는 모든 세이브에서 동일

## 6.5 도시 진출 방식

- 도시는 선형 순서로 해금하지 않음
- 조건을 충족한 도시 중 플레이어가 자유롭게 선택
- 각 도시는 프랜차이즈 매장 최대 1개
- 해당 도시의 매장을 철거·매각하면 그 도시만 게임 내 1개월 동안 재진출 불가
- 다른 도시와 국가는 영향받지 않음
- 개별 부지가 아니라 도시 자체를 선택
- 도시마다 초기 매장 크기, 인테리어 분위기, 기본 설비 배치가 다름

## 6.6 도시 데이터

각 도시는 다음 값을 가질 수 있다.

- 임대료
- 초기 진입 비용
- 생활비 수준
- 원재료 비용
- 직원 급여 수준
- 관광 수요
- 현지 고객 성향
- 가격 민감도
- 소비력
- 메뉴 카테고리 수요
- 경쟁도
- 경제 성장률
- 계절 민감도
- 재난·이벤트 위험도
- 추천 재조사 주기

대부분의 상세 값은 숨겨져 있으며 시장 조사를 통해 획득한다.

# 7. 시장 정보 시스템

## 7.1 기본 공개 정보

시장 정보를 구매하지 않은 도시에서도 다음은 공개한다.

- 매장 개설 비용
- 임대료
- 초기 매장 크기
- 생활비 등급
- 공개 경제 이벤트
- 공개 재난 이벤트
- 진입 조건
- `시장 정보 미확보` 경고

미조사 도시는 재조사 알림 개수에 포함하지 않는다.

## 7.2 정보 획득 방식

시장 상세 정보는 다음 경로로만 얻는다.

1. 암시장 정보상
2. 현지 식당 주인 자문
3. 특정 캐릭터의 개인 능력

무료 수동 시장 조사는 제공하지 않는다.

## 7.3 암시장 정보상

각 정보상은 고정된 개성을 가진다.

- 가격
- 정확도
- 전문 도시
- 전문 분야
- 사기 확률
- 제공 정보 수준

특성은 영구 고정되며 반복 거래로 신뢰도·할인·사기 확률이 변하지 않는다.

사기 발생 시:

- 지불 금액 전액 손실
- 정보 미획득
- 환불 없음
- 추적·처벌·복구 퀘스트 없음
- 기존 정보는 유지
- 이후 다른 이름으로 다시 등장할 수 있음

## 7.4 현지 식당 주인 자문

- 더 비쌈
- 사기 확률 0%
- 더 정밀하고 신뢰할 수 있는 현재 정보 제공

## 7.5 정보 저장

- 성공적으로 획득한 정보는 세이브에 유지
- 단, 변동 수치는 획득 당시의 스냅샷
- 도시별 가장 최근 성공 기록만 저장
- 과거 그래프나 이전 스냅샷 목록은 없음
- `확인 후 X일 경과`로 표시
- 도시별 추천 재조사 기간 존재
- 추천 기간이 지난 도시 수를 준비 화면의 알림 배지에 표시

## 7.6 등급 업그레이드 및 갱신

- 이전에 실제 지불한 성공 구매액은 상위 등급 구매 시 차감
- 사기로 잃은 금액은 할인에 포함하지 않음
- 캐릭터 능력으로 무료 획득한 정보는 할인 금액에 포함하지 않음
- 상위 등급 구매 시 도시 전체 항목을 최신 값으로 갱신
- 최고 등급 현지 자문은 할인된 갱신 비용을 제공할 수 있음
- 암시장 갱신은 계속 사기 위험 존재
- 인천 최초 무료 정보는 유료 누적액으로 계산하지 않음

도시와 국가의 경제는 다음에 따라 변화한다.

- 호황

- 관광 급증
- 재개발
- 공급 충격
- 자연재해

영향 대상:

- 장비 비용

- 소비력
- 가격 민감도

일반 변화는 점진적이며 큰 이벤트는 급격한 변화를 만들 수 있다.

## 8.2 경쟁자

경쟁 매장은 완전한 실제 매장 시뮬레이션이 아니라 추상화된 NPC 경쟁 값으로 처리한다.

- 처리 능력을 초과한 손님 유입
- 매장 평판 악화
- 잘못된 고객층 유입
- 재료 부족
- 대기 시간 증가

광고는 좋은 전략을 증폭하지만 잘못된 전략을 구제하지 않는다.

# 9. 대출 시스템

- 은행 대출 제공
- 활성 대출 최대 3개
- 총부채 한도는 신용도와 자산에 따라 결정
- 각 대출은 개별 원금·일일 이자·만기·만기 이자를 가짐
- 매 영업일 일일 이자를 자동 납부
- 일일 이자는 원금을 줄이지 않음
- 만기일에 원금 + 만기 이자 + 미납 이자 + 연체 비용 일괄 납부
- 부분 상환 불가
- 중도 상환 시 남은 원금 전체를 한 번에 상환
- 연체 시 경고·유예·신용 하락·대출 제한
- 은행이 자산을 자동 압류하지 않음
- 위기 시 플레이어가 직접 매장·장비·자산 매각 결정

큰 자산 매각의 공동 승인 규칙은 초기 수직 슬라이스 이후 결정 가능하도록 설계한다.

# 10. 직원 시스템

## 10.1 역할

직원은 고정 역할을 가진다.

- 전처리
- 조리
- 서빙
- 계산
- 청소
- 정비
- 매니저

초기 버전에서 직원은 직무를 변경할 수 없다.

## 10.2 채용

- 정규직만 제공
- 등급: D/C/B/A/S 계열
- 능력치와 특성은 채용 시 무작위로 고정
- 레벨업 없음
- 교육 성장 없음
- 등급 상승 없음
- 특성 변경 없음

## 10.3 최소 근무 기간

- 등급별 최소 고용일 존재
- 기간 전 해고 시 위약금·퇴직금
- 기간 이후에는 별도 위약금 없이 교체 가능

## 10.4 휴가 및 결근

- 직원별 매월 휴가 2~7일 무작위
- 실제 휴가 날짜는 무작위 지정
- 준비 단계에서 사전 확인 가능
- 휴가 날짜 이동·취소·매수 불가
- 휴가·질병·교육·결근·조퇴 중에도 급여 지급
- 개인 특성이 이벤트 패턴에 영향

## 10.5 재배치

- 준비 단계에서만 가능
- 거리 기반 운송 비용 지불
- 국가 간 이동은 더 비쌈
- 즉시 이동
- 임시 지원 근무 아님
- 매장 소속이 영구 변경
- 계약·휴가·최소 근무일 유지

## 10.6 작업 독점

직원이 설비에서 작업을 시작하면:

- 플레이어가 끼어들 수 없음
- 도울 수 없음
- 중단할 수 없음
- 아이템을 가져갈 수 없음
- 직원을 교체할 수 없음
- 주문이 취소돼도 작업은 끝까지 수행
- 결과물은 다른 동일 레시피 주문에 사용하거나 폐기 가능
- 직원과 플레이어 사이 작업 진행도 이전 불가

두 플레이어끼리는 같은 플레이어 작업을 협력할 수 있지만 직원 작업에는 합류할 수 없다.

# 11. 플레이어 캐릭터

## 11.1 캐릭터 선택

- 미리 제작된 고유 캐릭터
- 커스텀 캐릭터 생성 없음
- 세이브 생성 시 플레이어별 캐릭터 선택
- 같은 캐릭터 중복 선택 불가
- 세이브 내 캐릭터 변경 불가
- 새 세이브에서는 다른 캐릭터 선택 가능

## 11.2 캐릭터 구조

각 캐릭터는 다음을 가진다.

- 고유 외형
- 배경 설정
- 전문 분야
- 패시브
- 액티브 스킬
- 영구 업그레이드 트리

모든 캐릭터가 모든 작업을 수행할 수 있으며 전문 분야는 효율 차이만 만든다.

## 11.3 능력 범위

캐릭터 능력은 해당 캐릭터의 직접 행동에만 영향.

영향 금지:

- 전 세계 자동 버프
- 모든 직원 강화
- 모든 매장 강화
- 자동화 매장 수익 배율
- 다른 플레이어의 직접 행동

## 11.4 액티브 스킬

- 실시간 영업 중 자유 사용
- 고정 지속시간
- 수동 취소 불가
- 지속시간 종료 후 쿨다운 시작
- 준비·정산 단계에서는 시간 정지
- 층·매장 이동 중 상태 유지
- 재접속 시 상태 저장을 고려

## 11.5 성장

- 프랜차이즈 공용 자금 사용
- 캐릭터별 별도 영구 업그레이드
- 환불·초기화·재분배 없음

가능한 전문 분야:

- 굽기
- 전처리
- 서비스
- 이동·운반
- 청소
- 운영·시장 분석

# 12. 플레이어 인벤토리

## 12.1 기본 구조

- 마인크래프트식 슬롯 인벤토리
- 아이템 중첩 없음
- 슬롯 1개당 아이템 1개
- 기본 캐릭터 시작 슬롯 3개
- 최대 9개
- 잠긴 6개 슬롯은 처음부터 UI에 표시
- 준비 단계에서 공용 자금을 사용해 한 칸씩 영구 해금
- 캐릭터별 해금 상태 독립
- 한 캐릭터 확장이 다른 캐릭터에 영향 없음

## 12.2 운반 특화 캐릭터

- 시작 슬롯 4개
- 슬롯 확장 비용 할인
- 일반 최대치 9개
- 9개 모두 해금 후 숨겨진 최종 성장 노드 등장
- 해당 노드 구매 후 10번째 전용 슬롯 UI 표시
- 추가 비용으로 10번째 슬롯 해금
- 9개 이전에는 10번째 슬롯 자체가 보이지 않음

## 12.3 슬롯 선택

- 숫자 키
- 마우스 휠
- 선택 슬롯 아이템을 손에 표시

## 12.4 아이템 줍기

`J`로 아이템을 주울 때:

1. 현재 선택 슬롯이 비어 있으면 해당 슬롯에 저장
2. 선택 슬롯이 차 있으면 왼쪽부터 첫 빈 슬롯 사용
3. 잠긴 슬롯 제외
4. 빈 슬롯이 없으면 실패
5. 획득 후 선택 슬롯 자동 변경 없음

# 13. 입력 체계

## 13.1 기본 키

- 이동: WASD
- 달리기: 기존 설계와 프로젝트 입력 시스템 검토 후 결정
- `J`: 상호작용
- `K`: 조리
- `Q`: 선택 슬롯 아이템 바닥에 내려놓기
- 숫자 키·마우스 휠: 슬롯 선택
- `Shift + 클릭`: 인벤토리 빠른 이동

Unity의 새 Input System 사용을 우선 검토한다.

## 13.2 J — 상호작용

포함:

- 바닥 아이템 줍기
- 작업대 아이템 집기
- 작업대에 아이템 놓기
- 작업대 아이템 교환
- 설비에서 아이템 꺼내기
- 재료 보관함 사용
- 냉장고 열기
- 문·엘리베이터·일반 시설 사용
- 음식 제출

캐릭터 머리 위 안내는 항상:

`J: 상호작용`

상황별 상세 문구는 표시하지 않는다.

## 13.3 K — 조리

포함:

- 칼질
- 뒤집기
- 소스 섞기
- 튀김옷 입히기
- 기타 레시피 작업 진행

캐릭터 머리 위 안내는 항상:

`K: 조리`

행동별 세부 명칭은 표시하지 않는다.

---

# 14. 상호작용 대상 선택과 하이라이트

## 14.1 대상 우선순위

가까운 대상이 여러 개일 때:

1. 캐릭터가 바라보는 방향 우선
2. 같은 방향 후보 중 가장 가까운 대상
3. 전방에 없으면 주변에서 가장 가까운 대상

`J`와 `K` 모두 동일한 대상 선택 규칙 사용.

## 14.2 머리 위 안내

한 번에 가장 우선순위 높은 행동 하나만 표시.

조리 작업대 기준:

- 작업대가 비어 있음 → `J: 상호작용`
- 조리 가능한 아이템이 있고 작업 미완료 → `K: 조리`
- 조리 완료 아이템이 있음 → `J: 상호작용`

## 14.3 하이라이트

- 현재 선택된 대상 하나에 얇은 외곽선
- 대상 변경 시 즉시 이동
- 범위 이탈 시 제거
- 각 플레이어는 자신의 하이라이트만 봄
- 상대 플레이어 하이라이트는 보이지 않음
- 같은 대상 선택 시에도 각자의 로컬 화면에만 표시
- 하이라이트는 네트워크 동기화 불필요

---

# 15. 작업대와 설비 상호작용

## 15.1 일반 작업대

일반 작업대는 아이템 1개만 보유.

`J` 규칙:

- 작업대 비어 있음 + 선택 슬롯 아이템 있음 → 배치
- 작업대 아이템 있음 + 선택 슬롯 비어 있음 → 획득
- 둘 다 아이템 있음 → 즉시 스왑
- 둘 다 비어 있음 → 아무 동작 없음

스왑은 현재 선택 슬롯만 사용하며 다른 슬롯으로 자동 이동하지 않는다.

## 15.2 조리 설비

예:

- 도마
- 튀김기
- 버너
- 오븐
- 소스 작업대
- 튀김옷 작업대

작업 진행 중:

- 즉시 스왑 금지
- 작업 아이템을 꺼내려면 선택 슬롯이 비어 있어야 함
- 직원이 작업 중이면 모든 개입 금지

작업 완료 후:

- 스왑 허용
- 단, 새로 설비에 들어갈 아이템이 해당 설비 허용 조건을 통과해야 함
- 잘못된 아이템이면 전체 스왑 취소
- 양쪽 아이템 상태 유지

일반 작업대는 아이템 종류 제한 없이 스왑 가능.

## 15.3 실패 안내

허용되지 않는 아이템을 설비에 넣거나 스왑하려 하면:

- 인벤토리 UI 위에 안내
- 예: `이 아이템은 튀김기에 넣을 수 없습니다`
- 실패 효과음
- 2초 후 자동 제거
- 같은 오류 반복 시 2초 갱신
- 새 오류 발생 시 기존 문구 교체
- 오류 누적 표시 없음
- 아이템과 설비 상태 변화 없음

---

# 16. 바닥 아이템

## 16.1 기본 규칙

- 바닥 아이템은 오염되지 않음
- 자동 소멸하지 않음
- 플레이어와 직원이 위를 통과 가능
- 물리 장애물이 아님
- 조리 진행도·상태·품질 유지
- 다른 플레이어도 획득 가능
- 자동 획득 없음
- 가까이 가서 `J`로 획득

## 16.2 Q 배치

`Q` 입력 시 현재 선택 슬롯 아이템 1개를 배치.

배치 위치 우선순위:

1. 바라보는 방향 바로 앞 한 칸
2. 플레이어가 서 있는 바로 밑 칸
3. 플레이어 중심 3×3 범위의 다른 유효 바닥 칸

제외 대상:

- 벽
- 설비
- 가구
- 다른 바닥 아이템
- 이동 불가능 타일

모든 후보가 막히면 실패하고 아이템은 인벤토리에 유지.

## 16.3 1칸 1아이템

- 같은 바닥 칸에 아이템 중첩 불가
- 바닥 타일 하나당 최대 1개
- 사고로 떨어지는 아이템도 같은 규칙
- 유효한 낙하 위치가 없으면 낙하하지 않음

## 16.4 사고로 인한 낙하

- 미끄러짐·가벼운 넘어짐 → 현재 선택 슬롯 아이템 1개 낙하
- 지진·폭발·강한 밀침 → 보유 아이템 중 최대 2개 무작위 낙하
- 정전·연기·감전·일시 행동 불가 → 아이템 유지
- 층 이동·매장 이동 → 아이템 유지

## 16.5 동시 획득

- 호스트가 먼저 처리한 플레이어만 성공
- 다른 입력은 무시
- 별도 안내·실패 효과음 없음
- 즉시 최신 상태 동기화
- 중복 획득과 복제 방지

이와 같은 당연한 중복 방지·권한 처리 규칙은 별도 사용자 확인 없이 일관되게 적용한다.

# 17. 냉장고 시스템

## 17.1 목적

현재 버전에는 유통기한·신선도·부패 시스템이 없다.

냉장고의 목적은:

- 영업 마감 시 자동 폐기를 피함
- 다음 날까지 가열 전 재료를 보존
- 영업 중 임시 저장 공간 제공

## 17.2 보관 가능 상태

보관 가능:

- 생재료
- 손질된 재료
- 칼질 완료 재료
- 튀김옷 입힌 재료
- 기타 비가열 전처리 완료 재료

보관 불가:

- 한 번이라도 가열 조리를 시작한 재료
- 조리 중 음식
- 조리 완료 음식
- 과조리 음식
- 탄 음식

## 17.3 등급

기본 예시:

- 소형 냉장고: 3칸
- 중형 냉장고: 6칸
- 대형 냉장고: 9칸

정확한 가격과 수치는 ScriptableObject로 조정 가능하게 구성.

비싼 냉장고일수록 슬롯 수가 많다.

## 17.4 사용 시점

- 준비 단계 사용 가능
- 실시간 영업 중 사용 가능
- 냉장고를 열어도 세계 시간은 계속 흐름
- 주문·조리·직원·다른 플레이어·이벤트 계속 진행
- 냉장고를 연 플레이어만 이동과 조리 입력 제한

## 17.5 동시 사용

- 냉장고 하나는 한 번에 한 플레이어만 사용
- 먼저 연 플레이어가 사용권 획득
- 다른 플레이어는 열 수 없음
- UI 종료·연결 종료 시 사용권 자동 해제
- 내부 변경은 호스트 기준으로 즉시 동기화

## 17.6 UI 조작

마인크래프트식 슬롯 클릭 방식.

- 아이템 슬롯 클릭 → 커서에 아이템 들기
- 빈 슬롯 클릭 → 배치
- 아이템이 있는 슬롯 클릭 → 서로 교환
- `Shift + 클릭` → 반대편 인벤토리의 가장 왼쪽 유효한 빈 슬롯으로 빠른 이동
- 스택 없음
- 한 슬롯당 1개

교환 결과 냉장고에 들어갈 아이템은 보관 조건 검사.

불가능하면:

- 교환 취소
- 양쪽 아이템 원상 유지
- 2초 안내 문구
- 실패 효과음

## 17.7 UI 종료

냉장고 UI를 닫을 때 커서에 아이템이 있으면:

- 처음 집은 원래 슬롯으로 자동 복귀
- 아이템 분실·복제·바닥 낙하 금지
- 복귀 후 UI 종료

`Esc`, 거리 이탈, 연결 끊김, 강제 UI 종료 모두 안전하게 처리.

# 18. 마감 폐기 규칙

영업 마감 시 매장 맵 위 모든 아이템을 자동 폐기한다.

대상:

- 바닥
- 일반 작업대
- 도마
- 버너
- 튀김기
- 조리 설비
- 플레이어 인벤토리
- 조리 중 음식
- 완성 음식

예외:

- 냉장고 내부의 보관 가능한 가열 전 재료

폐기 결과는 정산 화면에서 수량 또는 비용으로 집계할 수 있게 설계한다.

# 19. 첫 메뉴와 조리 시스템

## 19.1 시작 메뉴

모든 세이브와 캐릭터 조합에서 동일:

- 기본 후라이드 닭강정

이후 연구로 추가:

- 달콤한 양념
- 매운 양념
- 간장
- 마늘
- 기타 지역별 변형

## 19.2 기본 조리 흐름

1. 생닭 획득
2. 도마에 배치
3. `K` 반복 입력으로 정해진 횟수만큼 칼질
4. 전용 튀김옷 작업대에서 `K` 한 번으로 전처리
5. 튀김기에 1인분 투입
6. 조리 완료 시 적절한 시점에 꺼내기
7. 용기에 담기
8. 주문 제출

## 19.3 칼질

- 정해진 반복 입력 횟수
- 진행도는 음식 오브젝트에 저장
- 중단 후 다른 플레이어가 이어서 가능
- 두 플레이어가 동시에 칼질 가능
- 모든 입력이 합산
- 협력 페널티 없음
- 직원 작업에는 플레이어 합류 불가

## 19.4 튀김

- 한 튀김기·바스켓에 1인분
- 초기 버전 배치 조리 없음
- 정확한 진행 바 표시
- 정상 완료 가능 구간 표시
- 초기 버전에서 정상 구간 폭은 레시피 공통
- 난이도는 조리 시간, 전처리, 장비, 주문량으로 차별화

## 19.5 조리 상태

- 덜 익음: 제출 불가, 꺼냈다가 다시 넣어 진행 가능
- 정상: 제출 가능
- 과조리: 제출 불가, 복구 불가, 폐기
- 탄 상태: 제출 불가, 폐기

## 19.6 완성 음식과 주문

- 음식은 특정 주문 ID에 귀속되지 않음
- 같은 레시피의 어떤 활성 주문에도 제출 가능
- 소스와 옵션이 다르면 별도 레시피
- 정상 완성 음식은 시간 경과로 식거나 품질 저하되지 않음
- 일반 작업대에는 음식 1개만 배치 가능
- 같은 메뉴라도 중첩 불가

- 공용 자금 + 연구 포인트 사용
- 구매 즉시 적용
- 선행 조건 존재
- 브랜드 수준
- 국가 진출
- 장비 해금
- 레시피 해금
- 운영 해금

연구 카테고리:

- 음식
- 조리 기술
- 장비
- 운영
- 물류
- 마케팅

연구는 도시의 숨겨진 수요나 수익성을 자동 공개하지 않는다.

# 24. 네트워크 권한과 동시성

## 24.1 호스트 권한

호스트가 최종 판정해야 하는 상태:

- 아이템 생성·이동·삭제
- 작업대 점유
- 설비 작업
- 냉장고 잠금
- 주문 생성·완료
- 직원 작업
- 재고
- 자금
- 대출
- 시장 정보
- 건설
- 매장 상태
- 저장

## 24.2 동시 상호작용

같은 대상에 두 플레이어가 동시에 접근하면:

- 호스트가 먼저 처리한 입력만 성공
- 대상 즉시 잠금
- 처리 후 상태 동기화
- 늦은 입력 무시
- 중복 획득·복제 방지
- 실패 플레이어에게 별도 메시지나 효과음 없음

## 24.3 로컬 전용 상태

네트워크 동기화 불필요:

- 상호작용 하이라이트
- 머리 위 키 안내
- 로컬 툴팁
- 카메라

# 38. 최종 핵심 요약

이 프로젝트의 최종 기준은 다음과 같다.

> 32×32 픽셀 타일을 사용하는 탑다운 2D 힐링 도트 게임이다.  
> 두 플레이어가 하나의 프랜차이즈를 공유하며 직접 요리하고, 직원과 여러 매장을 운영하고, 도시 시장을 분석하고, 국가를 확장한다.  
> 실시간 액션은 명확하고 빠르게 반응해야 하며, 장기 경영은 깊지만 UI와 아트는 따뜻하고 편안해야 한다.  
> 기존 3D 프로토타입은 필요한 로직만 추출하고 2D·데이터 중심·호스트 권한 구조로 재설계한다.  
> 가장 먼저 인천의 작은 매장에서 기본 닭강정을 만드는 완전한 2인 수직 슬라이스를 완성한다.
